## Project Overview

**ServerVault** is a web-based server management application built with Laravel, Vue, and Inertia.js. It helps users manage their servers in one centralized location with features like:

- User authentication and role-based access control
- Server inventory management
- Secure credential storage
- Browser-based SSH terminal
- Server health monitoring
- Audit logs

**Current Status:** Active development. Auth foundation, roles, permissions, and dashboard structure are complete. Main server management features are in progress.

## Architecture

### Monolithic Architecture with Inertia.js

ServerVault uses **Inertia.js** to bridge Laravel and Vue without building a separate REST API. Controllers return Inertia responses that render Vue components directly.

**Request Flow:**
```
Browser → Laravel Route → Controller → Inertia Response → Vue Component
```

**Key Points:**
- No separate API layer
- Vue components receive props from Laravel controllers
- Form submissions use Inertia's form helpers
- Route helpers generated by Ziggy for client-side routing
- WebSocket for real-time features (SSH terminal, monitoring)

### Layers

1. **Routes** (`routes/web.php`, `routes/api.php`)
2. **Controllers** (`app/Http/Controllers`)
3. **Form Requests** (`app/Http/Requests`)
4. **Models** (`app/Models`)
5. **Migrations** (`database/migrations`)
6. **Seeders** (`database/seeders`)
7. **Views** (`resources/js/Pages`)
8. **Components** (`resources/js/Components`)
9. **Layouts** (`resources/js/Layouts`)

## Component Development Rules

### General Rules

1. **Use Composition API only** — No Options API
2. **Use `<script setup>` syntax** for all Vue components
3. **TypeScript is disabled** — Use plain JavaScript
4. **Props validation** — Always define props with types
5. **Event naming** — Use kebab-case for emitted events
6. **Component naming** — PascalCase for file names and imports
7. **Single File Components** — Keep template, script, and style in one file

### Component Structure

Standard order:
```vue
<script setup>
// 1. Imports
import { ref, computed } from 'vue';

// 2. Props
const props = defineProps({
    modelValue: String,
});

// 3. Emits
const emit = defineEmits(['update:modelValue']);

// 4. Composables
const { theme } = useTheme();

// 5. Reactive state
const isOpen = ref(false);

// 6. Computed
const displayValue = computed(() => props.modelValue);

// 7. Methods
const handleClick = () => {
    emit('update:modelValue', 'new value');
};

// 8. Lifecycle hooks
onMounted(() => {
    // initialization
});
</script>

<template>
    <!-- Template -->
</template>

<style scoped>
/* Scoped styles (rare, prefer Tailwind) */
</style>
```

### Component Categories

#### 1. Custom Components (`resources/js/Components/`)

**Reusable, general-purpose components** used across multiple pages. Examples:
- `ApplicationLogo.vue`
- `ThemeToggle.vue`
- `OsIcon.vue`
- `FadeIn.vue`
- `MultiSelectFilter.vue`

**Rules:**
- Components should be **generic and reusable** across different pages
- Use shadcn-vue components as building blocks
- Apply theme-aware Tailwind classes
- Follow accessibility best practices
- Include proper ARIA attributes
- **Do NOT place page-specific modals here** (see Page-Specific Modals below)

#### 2. shadcn-vue Components (`resources/js/Components/ui/`)

Pre-built UI primitives from shadcn-vue. Examples:
- `Button.vue`
- `Dialog.vue`
- `Alert.vue`
- `Sidebar.vue`
- `Table.vue`
- `Select.vue`

**Rules:**
- **Do not modify directly** unless fixing a critical bug
- **Components/ui/ is ONLY for shadcn-vue components**
- Use class variants for customization
- Extend via composition, not modification
- Check shadcn-vue docs before changes
- Never add custom components here

#### 3. Layout Components (`resources/js/Layouts/`)

Page layouts. Examples:
- `AuthenticatedLayout.vue` — Main app layout
- `GuestLayout.vue` — Auth pages layout

**Rules:**
- Include header, navigation, and footer
- Use `<slot>` for page content
- Include theme toggle and user menu
- Handle responsive breakpoints

#### 4. Page Components (`resources/js/Pages/`)

Top-level views rendered by Inertia. Examples:
- `Dashboard.vue`
- `Users/Index.vue`
- `Servers/Index.vue`

**Rules:**
- Wrap in layout component
- Receive data via props from Laravel controllers
- Use `<Head>` component for page title
- Handle loading and error states

#### 5. Page-Specific Modals (`resources/js/Pages/{Resource}/Modals/`)

**Modals that belong to specific pages or features.** These are NOT reusable across the app.

**Location Pattern:**
```
resources/js/Pages/
├── Servers/
│   ├── Index.vue
│   ├── Modals/
│   │   └── ServerModal.vue        # Used only in Servers pages
│   └── Components/
│       └── ServerForm.vue         # Used only in Servers pages
├── Users/
│   ├── Index.vue
│   ├── Create.vue
│   ├── Edit.vue
│   ├── Modals/
│   │   └── UserModal.vue          # Used only in Users pages
│   └── Components/
│       └── UserForm.vue           # Used only in Users pages
```

**Examples:**
- `Pages/Servers/Modals/ServerModal.vue` — Create/Edit server modal
- `Pages/Users/Modals/UserModal.vue` — Create/Edit user modal
- `Pages/Users/Components/UserForm.vue` — User form component used in Create and Edit
- `Pages/Settings/Modals/ConfigModal.vue` — Settings configuration modal

**Rules:**
- Place modals in a `Modals/` subfolder within the related page folder
- Place page-specific components in a `Components/` subfolder within the related page folder
- Modal or component is specific to that feature/resource
- Import using relative paths: `import ServerModal from './Modals/ServerModal.vue'`
- Import page components: `import UserForm from './Components/UserForm.vue'`
- **Never place page-specific modals or components in the global `Components/` directory**
- If a modal or component is used across multiple pages/features, refactor it into a reusable component in the global `Components/` directory

#### 6. Page Components (Continued)

Top-level views rendered by Inertia. Examples:
- `Dashboard.vue`
- `Users/Index.vue`
- `Servers/Index.vue`

**Rules:**
- Wrap in layout component
- Receive data via props from Laravel controllers
- Use `<Head>` component for page title
- Handle loading and error states

### Styling Rules

#### **Never Use Hard-Coded Colors**

❌ **Bad:**
```html
<div class="bg-gray-900 text-white">
```

✅ **Good:**
```html
<div class="bg-background text-foreground">
```

#### **Use Theme-Aware Tokens**

Always use design tokens that adapt to light/dark mode:
```html
<!-- Backgrounds -->
<div class="bg-background">
<div class="bg-card">
<div class="bg-muted">

<!-- Text -->
<p class="text-foreground">
<p class="text-muted-foreground">
<span class="text-card-foreground">

<!-- Borders -->
<div class="border border-border">

<!-- Focus States -->
<button class="focus:ring-2 focus:ring-ring focus:ring-offset-2">
```

#### **Spacing & Layout**

Use Tailwind spacing scale:
- **Padding:** `p-4`, `px-6`, `py-2`
- **Margins:** `m-4`, `mx-auto`, `my-8`
- **Gap:** `gap-4`, `gap-x-2`, `gap-y-6`

Common patterns:
- **Card padding:** `p-6` or `p-8`
- **Section spacing:** `mb-6` or `mb-8`
- **Container:** `px-4 sm:px-6 md:px-8`

#### **Typography**

Use semantic font sizes:
```html
<!-- Headings -->
<h1 class="text-2xl font-semibold">Page Title</h1>
<h2 class="text-xl font-semibold">Section Title</h2>
<h3 class="text-lg font-medium">Subsection</h3>

<!-- Body -->
<p class="text-sm text-muted-foreground">Body text</p>
<span class="text-xs text-muted-foreground">Caption</span>
```

#### **Border Radius**

Use design token values:
- `rounded-lg` — Large radius (buttons, cards)
- `rounded-md` — Medium radius (inputs, tags)
- `rounded-sm` — Small radius (badges)

#### **Shadows**

Use sparingly:
- `shadow-sm` — Subtle elevation
- `shadow-md` — Cards and dropdowns
- `shadow-lg` — Modals
- `shadow-xl` — Popovers

#### **Transitions**

Always include transitions for interactive elements:
```html
<button class="transition-colors hover:bg-primary/90">
<div class="transition-transform hover:scale-105">
```

### Responsive Design

Use mobile-first breakpoints:
- `sm:` — 640px
- `md:` — 768px
- `lg:` — 1024px
- `xl:` — 1280px

Example:
```html
<div class="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-4">
```

### Dark Mode

Dark mode is controlled by the `dark` class on `<html>`:
```html
<html class="dark">
```

Theme is persisted in `localStorage` key: `server-vault-theme`

Values: `light`, `dark`, `system`

Use the `useTheme()` composable to manage theme state.

## Backend Conventions

### Controllers

**Location:** `app/Http/Controllers`

**Naming:** Singular resource name + `Controller` (e.g., `UserController`, `ServerController`)

**Methods:**
- `index()` — List resources
- `create()` — Show create form
- `store()` — Save new resource
- `show()` — Show single resource
- `edit()` — Show edit form
- `update()` — Update resource
- `destroy()` — Delete resource

**Return Types:**
- Use Inertia responses: `Inertia::render('Page', ['data' => $data])`
- Use type hints: `Response`, `RedirectResponse`
- Return JSON for API endpoints

**Example:**
```php
public function index(): Response
{
    return Inertia::render('Users/Index', [
        'users' => User::with('roles')->paginate(15),
    ]);
}
```

### Form Requests

**Location:** `app/Http/Requests`

**Naming:** Action + Resource + `Request` (e.g., `StoreUserRequest`, `UpdateServerRequest`)

**Structure:**
```php
class StoreUserRequest extends FormRequest
{
    public function authorize(): bool
    {
        return $this->user()->can('create users');
    }

    public function rules(): array
    {
        return [
            'name' => 'required|string|max:255',
            'email' => 'required|email|unique:users',
        ];
    }
}
```

### Resources (API Responses)

**Location:** `app/Http/Resources`

**Naming:** Resource + `Resource` (e.g., `UserResource`, `ServerResource`)

**Usage:** Transform models for API/Inertia responses

### Models

**Location:** `app/Models`

**Naming:** Singular, PascalCase (e.g., `User`, `Server`, `ServerDatabase`)

**Key Traits:**
- `use HasFactory;` — For seeding/testing
- `use HasUuids;` — For UUID primary keys
- `use SoftDeletes;` — For soft delete support
- `use HasRoles;` — For Spatie permissions

**Required Properties:**
```php
protected $fillable = []; // Mass-assignable fields
protected $hidden = []; // Hidden from JSON
protected $casts = []; // Attribute casting
protected $with = []; // Eager load relationships
```

**Relationships:**
- `hasMany()`, `belongsTo()`, `belongsToMany()`
- Use type hints for IDE support

### Migrations

**Location:** `database/migrations`

**Naming:** Auto-generated timestamp + description

**Rules:**
- Use `$table->uuid('id')->primary();` for UUID primary keys
- Use `$table->foreignUuid('user_id')->constrained();` for foreign keys
- Always include `up()` and `down()` methods
- Add indexes for frequently queried columns
- Use `nullable()` appropriately

**Example:**
```php
Schema::create('servers', function (Blueprint $table) {
    $table->uuid('id')->primary();
    $table->foreignUuid('user_id')->constrained()->onDelete('cascade');
    $table->string('name');
    $table->string('ip_address');
    $table->string('os');
    $table->timestamps();
});
```

### Seeders

**Location:** `database/seeders`

**Naming:** Resource + `Seeder` (e.g., `UserSeeder`, `RoleSeeder`)

**Run Order:** Defined in `DatabaseSeeder.php`

## Frontend Conventions

### Composables

**Location:** `resources/js/composables`

**Naming:** `use` + functionality (e.g., `useTheme`, `useModal`)

**Purpose:** Reusable stateful logic

**Example:**
```javascript
export function useTheme() {
    const currentTheme = ref('light');
    
    const setTheme = (theme) => {
        currentTheme.value = theme;
        localStorage.setItem('server-vault-theme', theme);
    };
    
    return { currentTheme, setTheme };
}
```

### Stores (Pinia)

**Location:** `resources/js/stores`

**Naming:** `use` + Store (e.g., `useAuthStore`, `useServerStore`)

**Usage:** Global state management

### Inertia Usage

#### Rendering Pages
```javascript
// Controller
return Inertia::render('Servers/Index', [
    'servers' => Server::paginate(15),
]);
```

#### Receiving Props
```vue
<script setup>
const props = defineProps({
    servers: Object,
});
</script>
```

#### Form Submissions
```vue
<script setup>
import { useForm } from '@inertiajs/vue3';

const form = useForm({
    name: '',
    email: '',
});

const submit = () => {
    form.post(route('users.store'));
};
</script>
```

#### Links
```vue
<Link :href="route('dashboard')" class="text-primary">
    Dashboard
</Link>
```

### Icons

Use **Lucide Vue Next**:
```vue
<script setup>
import { Server, Users, Settings } from 'lucide-vue-next';

<template>
```

**Rules:**

**Include in layout:**

## Database & Models

### Primary Keys

Use **UUIDs** for all models:
```php
use Illuminate\Database\Eloquent\Concerns\HasUuids;

class Server extends Model
{
    use HasUuids;
}
```

### Relationships

Common patterns:
```php
// User has many servers
public function servers(): HasMany
{
    return $this->hasMany(Server::class);
}

// Server belongs to user
public function user(): BelongsTo
{
    return $this->belongsTo(User::class);
}

// Server has many databases
public function databases(): HasMany
{
    return $this->hasMany(ServerDatabase::class);
}
```

### Enums

Use PHP enums for fixed values:
```php
enum ServerOs: string
{
    case Ubuntu = 'ubuntu';
    case Debian = 'debian';
    case CentOS = 'centos';
    case Windows = 'windows';
}
```

## Security & Permissions

### Authentication

- Uses **Laravel Breeze** with Inertia.js
- Email verification enabled
- Password confirmation required for sensitive actions
- Session-based authentication

### Authorization

Uses **Spatie Laravel Permission** package.

**Roles:**
- `superadmin` — Full access
- `admin` — User and server management
- `user` — Basic access

**Permissions:**
- `view users`, `create users`, `edit users`, `delete users`
- `view servers`, `create servers`, `edit servers`, `delete servers`

**Checking Permissions:**
```php
// Controller
$this->authorize('create', Server::class);

// Blade
@can('create users')
@endcan

// Vue (via props)
<template v-if="can.createServers">
```

**Middleware:**
```php
Route::middleware(['auth', 'role:admin'])->group(function () {
    //
});
```

## Common Patterns

### Modal Pattern

Use `Modal.vue` component:
```vue
<script setup>
import Modal from '@/Components/Modal.vue';
import { ref } from 'vue';

const showModal = ref(false);
</script>

<template>
    <Modal :show="showModal" @close="showModal = false">
        <!-- Modal content -->
    </Modal>
</template>
```

### Table Pattern

Use `@tanstack/vue-table`:
```vue
<script setup>
import { useVueTable } from '@tanstack/vue-table';

const table = useVueTable({
    data: props.servers,
    columns: [
        { header: 'Name', accessorKey: 'name' },
        { header: 'IP', accessorKey: 'ip_address' },
    ],
});
</script>
```

### Form Pattern

Use Inertia forms:
```vue
<script setup>
import { useForm } from '@inertiajs/vue3';

const form = useForm({
    name: '',
    ip_address: '',
});

const submit = () => {
    form.post(route('servers.store'), {
        onSuccess: () => form.reset(),
    });
};
</script>

<template>
    <form @submit.prevent="submit">
        <TextInput v-model="form.name" />
        <PrimaryButton :disabled="form.processing">
            Save
        </PrimaryButton>
    </form>
</template>
```

### Loading States

Handle Inertia loading:
```vue
<script setup>
import { router } from '@inertiajs/vue3';
import { ref } from 'vue';

const isLoading = ref(false);

router.on('start', () => isLoading.value = true);
router.on('finish', () => isLoading.value = false);
</script>
```

### Error Handling

Access validation errors:
```vue
<script setup>
import InputError from '@/Components/InputError.vue';

const form = useForm({
    email: '',
});
</script>

<template>
    <InputError :message="form.errors.email" />
</template>
```

### When Creating a New Feature

1. **Backend:**
   - Create migration: `php artisan make:migration create_resource_table`
   - Create model: `php artisan make:model Resource`
   - Create controller: `php artisan make:controller ResourceController`
   - Create form requests: `php artisan make:request StoreResourceRequest`
   - Add routes in `routes/web.php`
   - Run migration: `php artisan migrate`

2. **Frontend:**
   - Create page component in `resources/js/Pages/Resources/`
   - Create custom components in `resources/js/Components/`
   - Use shadcn-vue components from `resources/js/Components/ui/`
   - Add navigation link in `AuthenticatedLayout.vue`

3. **Permissions:**
   - Add permissions in `config/permission.php`
   - Seed permissions: `php artisan db:seed --class=RoleSeeder`
   - Check permissions in controllers and views

### When Styling Components

- ✅ Use `bg-background`, `text-foreground`, `border-border`
- ❌ Avoid `bg-gray-900`, `text-white`, `border-gray-200`
- ✅ Use `p-4`, `px-6`, `gap-4` for spacing
- ✅ Use `rounded-lg`, `rounded-md` for borders
- ✅ Add `transition-colors` to interactive elements
- ✅ Use responsive breakpoints: `sm:`, `md:`, `lg:`
- ✅ Use semantic font sizes: `text-sm`, `text-lg`

### When Working with Forms

- Use `useForm()` from `@inertiajs/vue3`
- Add CSRF token automatically via Inertia
- Handle validation errors via `form.errors`
- Show loading state via `form.processing`
- Use form requests in controllers for validation

### When Adding Permissions

- Define permissions in `config/permission.php`
- Seed via `RoleSeeder`
- Check in controllers: `$this->authorize('action', Model::class)`
- Check in views: Pass `can` array from controller
- Use middleware: `['auth', 'permission:view users']`
